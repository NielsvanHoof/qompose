# Qompose Platform

Multi-tenant platform for accounting and advisory firms to collect documents and answers from their clients. Firms build questionnaires per dossier (file uploads, text answers, yes/no confirmations), clients respond through a magic-link portal, and staff review, approve, or request corrections until the dossier is complete.

## Stack

- **Backend:** Laravel 13 (PHP 8.5), Fortify (auth, 2FA, passkeys), Spatie Multitenancy + Permission, activity log
- **Frontend:** Inertia v3 + React 19, Tailwind CSS 4, shadcn/ui, Wayfinder (typed routes)
- **Tooling:** Laravel Sail (Docker), Pest 4, Pint, Larastan, Biome + Prettier, Vite 8

## Getting started

Everything runs through Sail:

```bash
composer run setup          # install deps, .env, key, migrate, build assets
vendor/bin/sail up -d       # start Docker services
vendor/bin/sail composer run dev   # app + queue + Vite dev server
```

Useful commands:

```bash
vendor/bin/sail artisan test --compact   # backend tests (Pest)
vendor/bin/sail npm run types:check      # TypeScript
vendor/bin/sail npm run lint:check       # Biome (lint + import boundaries)
vendor/bin/sail npm run format:check     # Prettier
vendor/bin/sail bin pint --dirty         # PHP formatting
```

## Domain overview

- **Tenant / workspace** — a firm; all staff data is scoped per tenant.
- **Dossier** — a case for one client (`draft → awaiting_client → in_review → completed`).
- **Document request** — one questionnaire item on a dossier. Typed: `file`, `text`, or `boolean`. Status: `pending → submitted → accepted | rejected`.
- **Questionnaire template** — reusable set of items (system-wide or firm-owned) that can be applied to a dossier.
- **Client portal** — token-based magic-link page (no account) where clients answer requests.

## Frontend architecture

`resources/js` is organized by feature/domain:

```
resources/js/
├── pages/            Inertia entry points (thin, compose features)
├── layouts/          app, auth, and settings layouts
├── features/         domain code, one folder per domain
│   ├── document-requests/   item-type registry + types.ts
│   │   ├── staff/           staff-side cards and forms
│   │   └── portal/          client-portal renderers
│   ├── dossiers/  portal/  questionnaires/  clients/
│   ├── media/  dashboard/  workspaces/  security/  settings/
├── components/       shared only: ui/ (shadcn), app-shell/, sortable/, primitives
├── hooks/  lib/      shared hooks and utilities
├── types/            cross-cutting types only (auth, navigation, ui)
└── actions/ routes/  generated by Wayfinder — do not edit
```

`actions/` and `routes/` are gitignored. Run `vendor/bin/sail npm run dev` (or `npm run build`) so Wayfinder generates them — required for IDE type resolution and `npm run types:check`.

Key conventions (enforced by Biome `noRestrictedImports`; see `.cursor/rules/frontend-architecture.mdc`):

- Domain types live in `features/<domain>/types.ts`, not in the global `@/types` barrel.
- Shared `components/` never import from `features/`.
- Staff and portal renderers for document requests never import each other; the item-type registry (`features/document-requests/questionnaire-item-type-registry.tsx`) is the single place that wires both sides. Adding a new questionnaire item type means extending the `QuestionnaireItemType` union and adding one registry entry — TypeScript flags anything missing.

## Testing

Feature tests live in `tests/Feature` (Pest). Run the ones near your change:

```bash
vendor/bin/sail artisan test --compact tests/Feature/Dossiers/
vendor/bin/sail artisan test --compact --filter='portal'
```
